## 🎯 设计目标

### 核心目标
1. **全面性** - 整合所有本地信息源
2. **可发现性** - Agent 能快速找到所需信息
3. **关联性** - 建立知识之间的双向链接
4. **可持续性** - 易于维护和扩展
5. **标准化** - 统一的格式和规范

### 用户价值
- 新 Agent 5 分钟内了解全貌
- 快速定位项目信息和历史上下文
- 自动建立知识关联
- 支持定期回顾和知识沉淀

## 📁 目录结构设计

### PARA 方法 + Zettelkasten 混合架构

采用 **PARA** (Projects, Areas, Resources, Archives) 方法组织内容，结合 **Zettelkasten** 的永久笔记和双向链接理念。

#### 1. Projects (项目)
**原则**: 有明确目标和截止日期的任务集合

```
📁 01 - 项目中枢/
  📁 活跃项目/
    📁 项目名称/
      📝 项目总览.md      # 目标、背景、状态
      📝 技术架构.md      # 技术栈、架构图
      📝 开发日志.md      # 按时间记录进展
      📝 待办事项.md      # 当前任务
      📝 问题与解决.md    # 遇到的问题和方案
  📁 归档项目/
```

#### 2. Areas (领域)
**原则**: 需要长期维护的责任领域

```
📁 02 - 领域知识/
  📁 技术栈/           # 持续维护的技术能力
  📁 学习笔记/         # 持续学习的内容
  📁 最佳实践/         # 持续积累的经验
```

#### 3. Resources (资源)
**原则**: 感兴趣但暂时无需行动的参考材料

```
📁 03 - 资源库/
  📁 代码片段/         # 可复用的代码
  📁 工具配置/         # 配置文件和模板
  📁 参考资料/         # 参考文档和链接
```

#### 4. Archives (归档)
**原则**: 已完成或暂停的项目和领域

```
📁 04 - 归档/          # 归档内容
📁 05 - 外部信息/      # 待整理的外部信息
```

## 🔗 链接系统设计

### 双向链接策略

#### 1. 层级链接
建立从总览到细节的层级关系：
```markdown
# 在项目总览中
## 相关文档
- [[技术架构]]
- [[开发日志]]
- [[部署文档]]
```

#### 2. 关联链接
连接相关的概念和知识：
```markdown
# 在技术架构中
这个项目使用 [[Python]] 和 [[Docker]] 构建。
```

#### 3. 反向链接 (Backlinks)
Obsidian 自动显示引用当前笔记的其他笔记。

### MOC (Map of Content) 模式

每个主要领域都有一个 MOC 文件作为入口：

```markdown
# 项目索引 (MOC)

## 活跃项目
- [[airi/项目总览]]
- [[aipy/项目总览]]

## 归档项目
- [[旧项目/项目总览]]

## 按技术分类
### Python 项目
- [[aipy/项目总览]]

### Node.js 项目
- [[airi/项目总览]]
```

## 📝 内容创建工作流

### 信息流向

```
外部信息 → 收集箱 → 处理 → 项目/领域/资源 → MOC索引
                ↓
           临时笔记 → 永久笔记
```

### 工作流步骤

#### 1. 收集 (Capture)
- 快速记录到 `00 - 收集箱/`
- 使用简短、临时的笔记
- 不担心格式和组织

#### 2. 处理 (Process)
- 定期（每天/每周）清理收集箱
- 决定：项目？知识？资源？归档？
- 移动到相应位置

#### 3. 组织 (Organize)
- 完善笔记内容
- 添加 YAML Frontmatter
- 建立双向链接

#### 4. 回顾 (Review)
- 定期回顾 MOC 索引
- 发现缺失的链接
- 更新和优化知识结构
