# 安的个人书房总待办

这份待办用于把 `an-llm-wiki` 从“可看”推进到“可学习、可复刻、可维护”。
原则很简单：前端先像一间中文书房，内容先像给小白看的路书，后端只做公开数据的稳定出口，私有材料不进公开层。

## 0. 总原则

- 全部面向中文读者。
- 读者只能看到前端页面、站内详情页、公开 GitHub 或公开网站链接；不能假设读者会打开本地黑曜石、本地路径、后台 wiki 文件或维护文档。
- 前端详情页必须自包含：项目故事、复刻步骤、失败点、安的提醒、来源说明和下一步行动都要在网页里看得见。
- 藏馆用行动型中文大类，不直接暴露工程黑话；读者首先关心“我能靠 AI 做什么事”。
- 内容必须真实，不能靠空洞示例撑满页面。
- 知识内容禁止闭门造车，必须按 `docs/knowledge-content-source-and-review-standard.md` 走来源、适配、挑刺和验收。
- 公开层只放安全可讲的内容。
- 小安相关能力只走本地受控配置，不写入公开仓库。
- 安的书房要像展示柜，不像后台面板。
- 每个周期性自动化都必须声明三件事：`客观产物`、`Obsidian 可见落点`、`面向安的中文推送摘要`。

## 2.2 内容生产工作流

1. 来源调研：先找公开来源、个人一手记录、可引用截图或原始材料，顺手标注哪些能公开、哪些只能转成方法与结论。
2. 结构提炼：先出页面骨架，再判断它更适合放首页、藏馆、工坊、风信还是年谱；没有“读者看完能做什么”的结构就不进入写作。
3. 中文改写：把英文直译、工程黑话、维护者视角改成中文读者能直接理解的表达，每段都要回答一个具体问题。
4. 前端验收：实际检查卡片、详情页、移动端、来源区、行动按钮和更新时间展示，确认读者不依赖站外私有资料也能读懂。
5. 发布复核：最后再查事实错误、错链、公开边界、过度承诺、重复发布和更新时间，确认没问题再发布。

## 2.1 模块级内容标准

后期添加内容时，先判断模块，再按模块标准写入。任何模块都不能因为“页面空”就塞低质量内容。

| 模块 | 必须回答 | 必填内容 | 拒收规则 | 验收方式 |
|---|---|---|---|---|
| 首页 | 安是谁，书房是什么，新手从哪里开始 | 个人信号、核心入口、精选项目、最新风信、边界说明 | 技术词堆砌、营销口号、空泛自我介绍 | 首屏像中文个人书房，小白 10 秒知道去哪 |
| 藏馆 | 我能靠 AI 做什么事 | 行动型大类、适合谁、用途、风险、下一步行动 | 英文技术分类直出、只有工具名、无使用场景 | 卡片能让读者决定是否点开 |
| 谱系 | 我怎么一步步复刻 | 阶段、前置条件、交付物、检查点、失败处理 | 只有目录、没有完成标准、步骤不可执行 | 新手能照着做出最小版本 |
| 风信 | 发生了什么，安的判断怎样变化 | 信息来源、趋势判断、认知变化、建议行动 | 纯新闻搬运、无判断、无下一步 | 读者能知道为什么这条值得看 |
| 工坊 | 安做了什么，别人怎样复刻 | 背景、工具材料、真实操作、复刻步骤、失败点、安的提醒 | 卡片代替详情、只展示结果、没有失败和边界 | 详情页能独立教会一个小项目 |
| 手记 | 这段经历留下什么判断 | 具体事件、心理层、社会层、哲学层、方法修正 | 无事件支撑的抒情、空洞鸡汤、私人原文 | 读完能看到一次判断如何形成 |
| 年谱 | 真实时间线上发生了什么 | 日期、阶段、事件、结果、相关页面、后续影响 | 虚构时间、用模板填空、没有来源 | 2026 年 3 月后的成长线可追踪 |
| 小安 | 它怎样服务读者 | 人格边界、可问范围、不能说什么、引用公开资料 | 暴露访问材料、装作知道私密事实、英文回复 | 对话中文、克制、有边界、有下一步 |

通用最佳实践：

- 用 GitHub 技能链生产内容：`data-extractor/doc-parser` 提取材料，`deep-research/academic-search` 补外部依据，`content-writer` 写成教程，`brand-guidelines` 对齐安的书房气质，`ux-writing-skill` 打磨前端可见文案，`seo-optimizer` 检查标题、摘要和可发现性。
- 用 Diátaxis 分清：教程、操作指南、参考资料、解释，不要混成一团。
- 用信息架构原则分组：按读者心智和行动任务分类，不按内部文件夹分类。
- 用公开安全原则发布：敏感材料只转化为方法、边界和复刻经验，不发布原文。
- 用前端验收反推写作：卡片、详情页、搜索、筛选、移动端都能自然展示，才算写完。
- 用挑刺审查防止闭门造车：每个重要知识条目都要检查来源不足、过度概括、小白难读、隐私泄露和复刻不可执行。

## 3.1 数据生成层真实问题检查清单

| 优先级 | 检查项 | 具体判断 | 处理动作 |
|---|---|---|---|
| 高 | 正文缺失却进入前端 | 详情只有摘要、没有正文段落、没有行动建议或没有来源区 | 直接阻断该条公开输出，回到内容生产补正文 |
| 高 | 来源字段是假来源 | `sourceLinks` 为空、只剩本地路径、只写“见笔记”、只挂后台文件名 | 直接退回，不允许前端上线 |
| 高 | 详情页仍依赖站外私有材料 | 文案里出现“去看 Obsidian / 去看本地文件 / 去看 wiki 原文 / 联系我拿资料” | 直接标记为不合格内容，必须改成站内自包含说明 |
| 高 | 中文分类映射失效 | 前端仍读到英文 tag、原始技术分类、内部枚举名 | 阻断发布，先修分类映射表和前端展示字段 |
| 高 | `hiddenReasons` 与可见状态冲突 | 条目已被判低质或高风险，但 `visible` 仍为真 | 视为发布级错误，生成阶段直接失败 |
| 高 | 关联页面断链 | 风信、年谱、工坊引用的 slug、id 或 related links 在公开数据里不存在 | 生成报告标红，阻断该条或整批发布 |
| 中 | `actionText` 为空或是空话 | 只有“敬请期待”“持续关注”这类句子，没有可执行下一步 | 不进首页精选，退回补具体动作 |
| 中 | 年谱日期不可用 | 只写“最近”“前段时间”，没有明确日期或至少明确到月份 | 不生成年谱公开条目 |
| 中 | 风信结构不完整 | 没有事实、没有判断、没有下一步行动三者之一 | 不进入风信列表，退回重写 |
| 中 | 工坊缺复刻核心字段 | 没有材料、步骤、失败点、复刻检查任一项 | 不进入工坊详情公开层 |
| 中 | 更新时间失真 | 内容已改，但 `updatedAt` 没更新，导致 App 无法判断新旧 | 生成阶段强制刷新时间戳并写入质量报告 |
| 中 | 重复 slug 或重复卡片 | 两条内容 slug 相同，或同一内容以不同 id 重复生成 | 生成阶段直接失败，人工确认保留哪一条 |
| 低 | 封面与内容不匹配 | 卡片封面和正文主题无关，导致首页和列表误导读者 | 记入质量报告，发布前替换资产 |
| 低 | 深度字段空转 | 心理层、社会层、哲学层存在字段但没有证据支撑 | 不阻断发布，但不能放入精选或详情重点区 |

## 8. 验收标准

- 首页像“安的个人书房”，不是后台；首屏 10 秒内要让读者知道从哪开始。
- 读者不需要知道本地黑曜石在哪里，也不需要打开 GitHub 文件树，前端详情页本身就能读懂、判断和复刻。
- 藏馆分类是中文大类，不是工程黑话；卡片上能直接看出解决什么问题、适合谁、下一步做什么。
- 工坊页能让新手看懂怎么复刻，并能在页面内看到失败点、检查点和来源说明。
- 风信页能让读者知道发生了什么、为什么重要、下一步做什么，并能跳到相关工坊、藏馆或年谱。
- 生成的数据能通过隐私、结构、断链、时间戳和分类映射检查。

### 8.1 小安对话验收

- 全程中文回答，不装客服，不转英文模板。
- 先回答公开可说的内容，再明确边界；遇到私密材料、访问来源、内部配置时必须拒答并解释原因。
- 回答里至少给一个下一步动作，例如指向站内详情、公开项目页或可执行问题。
- 不编造“安已经做过”的事实；引用站内公开内容时，能说清来自哪一类页面。

### 8.2 App 实时更新验收

- 公开内容发布后，网页端与已安装 App 都能在可接受时间内拿到新数据。
- App 至少覆盖“首次打开、手动刷新、回到前台”三种更新场景，并能显示最后更新时间。
- 更新失败时要保留上一次可用内容，并给出可理解的提示，不出现空白页。

### 8.3 站内详情自包含验收

- 任何详情页都不能把关键说明外包给本地文档、Obsidian、后台 wiki 或私聊材料。
- 工坊详情至少有背景、步骤、失败点、检查点、来源、下一步行动。
- 风信详情至少有事实、判断、行动、来源和相关页面链接。
- 年谱详情至少有日期、事件、结果、后续影响和可回指页面。

## 9. 小白内容升级待办

这部分只管一个目标：让第一次进来的人不懂代码也能看懂，也知道下一步做什么。

### 9.1 三层读者标准

| 层级 | 读者问题 | 页面写法 | 通过标准 |
|---|---|---|---|
| 小白 | 我能不能靠 AI 做成一件事 | 先讲用途，再讲最小步骤，再讲常见坑 | 不认识框架名也能看懂 |
| 极客 | 我该怎样组合工具和流程 | 补方法、边界、替代方案、数据流 | 能据此搭出自己的版本 |
| 码神 | 我怎样继续优化架构和体验 | 补实现细节、约束、扩展点、验收 | 能看出设计取舍和升级空间 |

规则：

- 首页、藏馆卡片、谱系入口默认先服务小白。
- 详情页正文顺序固定为“小白先读 -> 极客延伸 -> 码神补充”。
- 任何页面如果一开头就在讲框架名、库名、模型名、协议名，视为不合格。

### 9.2 模块级补强清单

| 优先级 | 模块 | 要补什么 | 最低完成量 |
|---|---|---|---|
| P0 | 首页 | 首屏一句话、起步入口、安的自我介绍、读者边界 | 1 个首屏、3 个起步入口、1 段安的引导 |
| P0 | 藏馆 | 中文大类、卡片用途、适合谁、下一步行动 | 10 张高质量卡片 |
| P0 | 工坊 | 复刻步骤、失败点、检查点、安的提醒 | 3 个完整资料包 |
| P0 | 谱系 | 从零到做出成果的路线 | 2 条路线 |
| P1 | 风信 | 事实、判断、行动 | 8 条高质量风信 |
| P1 | 年谱 | 2026 年 3 月后的真实节点 | 12 个关键节点 |
| P1 | 小安 | 能问什么、不会答什么、怎样继续阅读 | 1 套公开问答边界页 + 站内入口 |
| P2 | 手记 | 情绪、判断、方法修正的中文表达 | 4 篇高质量手记 |

### 9.3 资料包标准

以后每一个卡片背后都要对应一个完整资料包，不能只有几句摘要。

每个资料包必须包含：

1. 这件事是什么
2. 为什么值得做
3. 适合什么人
4. 最小完成版本
5. 准备材料
6. 具体步骤
7. 常见失败点
8. 复刻检查清单
9. 安的提醒
10. 延伸阅读或公开来源

拒收情形：

- 只有结果图，没有步骤。
- 只有观点，没有证据。
- 只有工具列表，没有任务场景。
- 一上来就堆英文术语。
- 不能让读者完成最小版本。

### 9.5 小白验收问题

每次上线前，至少拿这 8 个问题过一遍：

1. 读者第一次进来，10 秒内知道这里是什么吗。
2. 他不懂前端、不懂后端，也知道第一步该点哪里吗。
3. 卡片上有没有只属于维护者的黑话。
4. 详情页有没有把关键说明外包给站外私有材料。
5. 资料包有没有最小可复刻版本。
6. 有没有把失败点写出来，而不是只展示成功结果。
7. 页面语言是不是中文、克制、清楚，而不是技术翻译腔。
8. 读完后有没有一个真实的下一步动作。

## 11. 自动化自我迭代节奏

当前自动化分三段，不追求一次做完全部，而是每天稳定推进：

1. `安的书房·风信趋势更新`
   - 早上运行
   - 抓最新 AI 高信号变化
   - 写成“事实 + 安的判断 + 下一步行动”
   - 优先回写风信候选、补缺建议和公开数据

2. `安的书房·内容补完迭代`
   - 下午运行
   - 只挑一个最高杠杆内容缺口
   - 优先补 首页、藏馆、工坊、谱系、风信、年谱、小安边界文案
   - 强制按“小白先读 -> 极客延伸 -> 码神补充”组织正文

3. `安的书房·质量巡检与回写`
   - 晚上运行
   - 查正文缺失、来源失真、英文外露、断链、更新时间失真、安装提示异常等问题
   - 先修一个最关键低风险问题，再把其余问题回写待办

自动化原则：

- 每次只解决一个最关键问题，不摊大饼。
- 先有证据，再有判断，再写内容。
- 能补正文就不只写摘要。
- 能站内讲清楚，就不把关键说明外包到后台资料。
- 不泄露私有材料、密钥、本地路径和未公开上下文。
