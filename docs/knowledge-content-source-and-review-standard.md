# 知识内容来源与审查标准

这份标准回答一个问题：安的书房里的知识内容不能闭门造车，怎样才算有依据、有审查、符合安的个人气质，也适合中文小白读者。

本标准适用于公开 `wiki/`、`site-data/`、前端展示文案、小安提示词、教程、工具卡、风信、工坊、年谱和后续 App 内容。私有材料可以作为证据来源，但进入公开层前必须改写、脱敏和复核。

## 1. 核心原则

任何知识内容都必须同时经过四个判断：

1. **证据判断**：这件事来自哪里，是安真实做过、官方文档说明、公开项目案例，还是经过复核的外部资料。
2. **适配判断**：这个来源为什么适合安的书房，为什么适合中文新手，而不是只适合工程师。
3. **反方判断**：它可能误导在哪里，有哪些边界、失败条件、替代方案和不能直接复制的地方。
4. **表达判断**：写出来之后是否有文学质感、心理层、社会层和哲学层，但这些层必须依附真实事件和操作。

个人经历可以成为证据，但不能单独变成通用结论。外部最佳实践可以成为方法，但不能替代安自己的上下文。

还有一条前端硬约束：读者只能看到网页、未来 App、站内详情页、公开 GitHub 链接和公开网站链接。任何内容都不能以“读者去本地黑曜石看原文”“读者打开后台 wiki 文件”“读者知道本机路径”为前提。后台页面只负责生产数据，前端详情页必须把该讲的内容讲完整。

## 1.1 GitHub 技能链

内容生产必须优先使用 GitHub 技能和本项目模块技能组合，不靠临场发挥。

已安装并纳入流程的 GitHub 技能：

| 技能 | 来源 | 用途 |
|---|---|---|
| `content-writer` | `claude-office-skills/skills` | 把材料写成文章、教程、操作指南和长文结构。 |
| `deep-research` | `claude-office-skills/skills` | 做多角度研究、来源综合、反方分析和建议。 |
| `academic-search` | `claude-office-skills/skills` | 涉及心理学、社会学、哲学、认知负荷等概念时，用于文献检索策略和研究框架。 |
| `brand-guidelines` | `claude-office-skills/skills` | 固化“安的个人书房”和“小安”的视觉、语气、人格与表达边界。 |
| `seo-optimizer` | `claude-office-skills/skills` | 检查标题、摘要、页面结构、站内链接和可发现性。 |
| `ux-writing-skill` | `content-designer/ux-writing-skill` | 打磨按钮、空状态、错误提示、卡片说明和小白可读文案。 |
| `data-extractor` | `claude-office-skills/skills` | 从资料、文档和混合来源中提取结构化事实。 |
| `doc-parser` | `claude-office-skills/skills` | 解析复杂文档、表格、章节和结构。 |

标准内容链：

```text
资料候选
  -> data-extractor / doc-parser 提取事实
  -> deep-research / academic-search 补外部依据和反方视角
  -> archive-content-curator 做公开安全改写
  -> content-writer 写成中文教程、项目故事或资料卡
  -> brand-guidelines 对齐安的书房气质
  -> ux-writing-skill 打磨前端可见文案
  -> seo-optimizer 检查标题、摘要和可发现性
  -> 对应模块技能检查
  -> 隐私、结构、前端验收
```

没有经过这条链的重要内容，只能进入候选队列，不能进入首页精选、藏馆推荐、工坊核心卡、风信主列表和小安公开上下文。

## 2. 来源组合

不同内容需要不同来源组合。低于最低组合的内容不能进入前端核心展示。

| 内容类型 | 最低来源组合 | 必须说明 |
|---|---|---|
| 工坊项目 | 安的公开安全项目证据 + 至少 1 个官方或行业最佳实践 + 1 次审查挑刺 | 做了什么，怎样复刻，哪里会失败，哪些内容被脱敏 |
| 藏馆资料 | 官方文档或公开项目 + 安的使用场景 + 小白下一步行动 | 适合谁，用来做什么，风险是什么 |
| 谱系路线 | 至少 1 个真实项目链路 + Diátaxis 或教程设计依据 + 验收标准 | 阶段、交付物、检查点和失败处理 |
| 风信 | 外部信息或趋势来源 + 安的判断变化 + 建议行动 | 发生了什么，为什么值得看，读者下一步做什么 |
| 手记 | 具体事件 + 方法变化 + 三层反思 | 不是散文，不是鸡汤，必须能看出判断如何形成 |
| 年谱 | 真实日期或可解释日期 + 相关公开页面或来源说明 | 不能虚构时间，不能用文件修改时间冒充事件 |
| 概念解释 | 至少 2 个可信来源，优先官方、论文、标准或成熟框架 | 不确定就写不确定，不能把个人理解写成事实 |
| 小安回答规则 | 人格边界 + 安的公开资料 + 安全规则 + 提示词评审 | 中文、克制、服务读者，不暴露访问材料 |

## 3. 推荐来源层级

优先级从高到低：

1. 官方文档、标准、论文、权威机构指南。
2. 成熟开源项目、GitHub 示例、官方仓库 issue 或 discussion。
3. 安自己的公开安全项目记录、复刻过程、浏览器验收、测试结果。
4. 私有材料的脱敏摘要、文件清单、哈希、来源说明。
5. 个人判断和经验总结。

第五层只能补充解释，不能单独作为事实依据。

## 4. 心理学、社会学、哲学的写法

三层理解不是装饰，必须落到具体材料。

心理学层回答：读者或安在这件事里遇到了什么注意力、动机、焦虑、拖延、选择困难或学习摩擦。不能写诊断，不能给人贴标签。

社会学层回答：工具、平台、学校、社区、市场、协作方式和公开展示环境怎样影响了这件事。不能空谈时代，不把平台当神话。

哲学层回答：这件事怎样触及行动、边界、责任、失败、时间、手艺和人的主动性。要短，要克制，要能回到项目证据。

通过标准：删掉这三层之后，页面仍然有完整事实；加上这三层之后，读者更懂为什么这件事值得学。

## 5. 子代理与挑刺规则

涉及知识内容、前端主模块、后端接口、小安人格、公开教程、趋势判断时，必须至少有一次挑刺步骤。可以由子代理、代码审查技能、人工 checklist 或测试报告完成。

挑刺必须问：

1. 这条内容有没有把个人经验冒充普遍真理。
2. 这条内容有没有缺少官方或公开依据。
3. 这条内容有没有对小白不友好，使用过多黑话。
4. 这条内容有没有泄露访问材料、本地路径、账号状态或私有上下文。
5. 这条内容有没有只有文学表达，没有可执行步骤。
6. 这条内容有没有只有技术步骤，没有心理、社会和哲学层的解释。
7. 这条内容有没有让前端展示变复杂、分类变混乱或移动端难读。

如果不能使用子代理，就必须在页面或质量记录中写明手动挑刺结论。

## 6. 前端展示标准

前端是给读者看的，不是给资料库维护者看的。

必须做到：

- 详情页自包含，不依赖本地资料库、后台资料层、维护文档或本机文件。
- 读者可点击的来源只使用站内详情、公开 GitHub、官方文档和公开网站；不展示本机盘符路径、后台资料路径或 Obsidian 双链。
- 中文优先，不把英文枚举、内部标签和工程目录名作为主展示。
- 卡片回答“这是什么、适合谁、为什么看、下一步做什么”。
- 详情页回答“怎样复刻、哪里会失败、安提醒什么”。
- 风信固定呈现“外部变化、安的判断、建议行动”。
- 工坊卡片只做入口，完整内容在详情页。
- 图片和封面要服务主题，不用同一张图铺满所有模块。
- 空状态比低质量填充更好。

## 7. 公开安全边界

永远不公开：

- 任何真实鉴权材料、浏览器会话材料、远程登录材料、真实中转站访问材料。
- 本机绝对路径、服务器登录细节、环境变量值、账号状态。
- 原始聊天记录、私密记忆、未复核提示词、私人文件原文。
- 不能判断安全的本地材料。

可以公开：

- 方法、结构、复刻步骤、失败模式、处理原则。
- 脱敏后的项目故事、工具路线、操作记录。
- 官方文档和公开项目的引用与改写。
- 安的成长线，但要保留真实时间和必要边界。

## 8. 页面元数据建议

高价值公开页建议补充这些 frontmatter 字段：

```yaml
sourceLabels: [public wiki, source-backed]
publicSafety: public-safe
evidenceLevel: source-backed
referenceTypes: [personal-evidence, official-docs, best-practice, review]
reviewStatus: challenged
reviewNotes: "已检查来源、隐私、复刻性和中文小白可读性。"
audience: "中文 AI 新手、未来的安、协作者"
```

字段不完整时可以先不上架，或标记为 `starter`、`needs-source-review`、`needs-redaction`。

## 9. 发布前验收

任何知识内容进入前端核心展示前，至少通过：

1. 有来源，且来源类型写清楚。
2. 有中文标题、摘要、下一步行动。
3. 有证据、操作、复刻、失败、边界。
4. 工坊和重要教程有心理层、社会层、哲学层。
5. 有挑刺记录或审查结论。
6. 隐私扫描不报高风险。
7. 前端详情页能自然展示。
8. 前端页面不要求读者访问本地黑曜石、本地路径、私有仓库文件或后台生成数据。

不满足时，保留在私有层、复核队列或公开 wiki 非核心页，不进入首页、藏馆、工坊精选和 App 主路径。

可执行闸门：运行 `python scripts/check_public_content_quality.py .`。它会阻断精选页、生命周期学习包和前端晋升内容里的硬问题；普通 curated 页的改进项会进入 `manifests/public_content_quality_report.csv`，用于分批提升。

## 10. 当前参考框架

- Karpathy LLM Wiki：原始资料、编译 wiki、schema 分层。
- Obsidian Properties：frontmatter、标签、别名、日期和结构化属性。
- Diátaxis：教程、操作指南、参考、解释分离。
- NN/g：用用户语言和真实世界心智模型组织导航。
- 认知负荷理论：给新手减少无关负担，提供步骤、例子和验收。
- WCAG：可访问、可理解、可操作、结构清楚。
- React 与 PWA 官方实践：数据流清楚、缓存和更新策略明确。
- OWASP：真实鉴权材料和会话类材料永远不进前端和公开仓库。
