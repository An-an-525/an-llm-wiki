# 前端内容生产完整工作流

本工作流用于安的个人书房网站和未来 App。核心原则：读者只能看到前端页面、站内详情页、公开 GitHub 链接和公开网站链接。后台黑曜石、本地路径、私有层、生成脚本和维护文档只能服务生产，不能成为读者理解内容的前置条件。

## 1. 工作目标

每一条进入前端的内容，都必须让中文小白读者在网页里完成三件事：

1. 看懂这是什么；
2. 明白它为什么值得看；
3. 知道怎样复刻一个更小版本。

如果一条内容只能在后台文档里看懂，前端就还没有完成。

## 2. 标准技能链

每批内容按以下技能链执行：

| 顺序 | 技能 | 作用 |
|---|---|---|
| 1 | `data-extractor` / `doc-parser` | 从本地资料、公开网页、文档或项目说明中提取事实、动作、时间、工具、产物。 |
| 2 | `deep-research` | 查找多方公开来源，形成背景、反方、风险和最佳实践。 |
| 3 | `academic-search` | 涉及心理学、社会学、哲学、认知负荷、学习方法时，提供研究型检索框架。 |
| 4 | `archive-content-curator` | 做公开安全改写，删除私密细节、访问材料、本机路径和原始聊天。 |
| 5 | `content-writer` | 写成中文教程、项目故事、资料卡、风信、手记或学习路线。 |
| 6 | `brand-guidelines` | 对齐“安的个人书房”：文艺、克制、清楚、有人味，不写营销腔。 |
| 7 | `ux-writing-skill` | 打磨卡片、按钮、空状态、错误提示、详情页导语和小安回复。 |
| 8 | `seo-optimizer` | 检查标题、摘要、分节、站内链接和公开可发现性。 |
| 9 | 模块技能 | 首页、藏馆、谱系、风信、工坊、手记、年谱、小安分别过模块准入。 |
| 10 | 安全与前端验收 | 跑隐私、结构、数据、lint、build 和浏览器检查。 |

## 3. 输入分层

输入材料分四类处理：

| 输入 | 处理方式 | 是否直接进前端 |
|---|---|---|
| 安的公开安全项目事实 | 提取、改写、补来源、写成详情页 | 可以 |
| 官方文档、GitHub 项目、公开教程 | 摘要、引用、转化为小白说明 | 可以 |
| 非公开资料、聊天、个人线索 | 只提取脱敏方法、时间线和经验 | 不能直接进入 |
| 敏感访问材料、账号、路径、服务器细节 | 只记录安全原则和处理状态 | 永不进入 |

## 4. 模块输出标准

### 首页

必须让读者知道：安是谁，小安是什么，这间书房能帮他做什么，应该从哪里开始。

### 藏馆

每张资料卡必须回答：

- 这是什么；
- 适合谁；
- 能帮你做什么；
- 为什么安收进书房；
- 风险或边界；
- 下一步行动。

主分类只能用中文行动大类：做前端、做后端、用工具、搭智能体、问小安、写提示词、整理资料、走学习路线、保护隐私、看参考资料。

### 谱系

每条路线必须包含阶段、前置条件、交付物、检查点、失败信号和修复动作。读者要能按它做出一个小作品。

### 风信

每条风信必须包含：

- 外部变化或资料库变化；
- 安的判断怎样改变；
- 读者下一步做什么。

### 工坊

每个项目必须有：

- 项目背景；
- 真实操作历程；
- 工具和材料；
- 小白复刻路径；
- 失败模式；
- 安的提醒；
- 心理层、社会层、哲学层；
- 当前状态和下一步。

### 手记

手记可以有文学性，但必须落在具体事件和方法修正上，不能写成空泛感想。

### 年谱

年谱按真实时间写。安基本从 2026 年 3 月之后开始系统玩 AI，不能用虚构日期填空。

### 小安

小安是安的数字生命体，也是这间书房的整理者和导览者。小安不是普通客服，也不是通用聊天机器人。它帮助安把散乱材料编译成读者能看懂、能复刻、能长期学习的中文书房。

对安工作时，小安可以读取本地上下文来整理材料、发现缺口、规划模块和改进前端。对读者展示时，小安只使用公开资料和用户当前问题。小安不暴露访问材料、本机路径、非公开记忆、服务器细节或未公开事实。

## 5. 写作标准

一条高质量内容必须同时具备：

- 真实证据；
- 可复刻步骤；
- 失败点；
- 安的提醒；
- 文学质感；
- 心理层；
- 社会层；
- 哲学层；
- 公开边界；
- 前端可见正文。

文学性不是堆形容词。它应该让句子更准确、更有温度、更能记住。心理学、社会学和哲学也不是装饰，必须服务具体操作。

## 6. 前端自包含标准

禁止在前端中出现这些读者指引：

- “去本地资料库查看”；
- “打开本机盘符路径”；
- “查看 `wiki/...` 文件”；
- “查看非公开层 / 原始层 / 生成脚本”；
- “查看本地 manifest 才能理解”。

允许出现：

- 站内详情页；
- GitHub 公开仓库；
- 官方文档；
- 公开项目；
- 公开网站；
- 已改写到当前页面里的来源说明。

## 7. 审核清单

每条内容进入前端核心展示前，必须回答：

1. 读者不看后台，能不能理解？
2. 读者能不能照着做出最小版本？
3. 有没有明确失败点和恢复方式？
4. 有没有把个人经验冒充普遍真理？
5. 有没有泄露敏感访问材料、路径、账号、服务器、原始聊天？
6. 有没有英文工程标签压过中文行动语言？
7. 心理、社会、哲学三层是否依附真实事件？
8. 卡片、详情、搜索、筛选、移动端是否都能自然展示？

## 8. 验收命令

从 vault 根目录运行：

```powershell
python scripts/wiki_check.py .
python scripts/privacy_scan.py .
python scripts/check_public_content_quality.py .
python scripts/build_site_data.py .
python scripts/build_public_inventory.py .
python -m unittest discover -s tests
git diff --check
```

前端改动后运行：

```powershell
cd site
npm run lint
npm run build
```

浏览器至少检查：

- 首页；
- 藏馆；
- 谱系；
- 风信；
- 工坊；
- 年谱；
- 小安对话；
- 至少三个详情页。

## 9. 当前优先级

1. 让前端详情页自包含，不依赖本地黑曜石。
2. 清理前端数据里的后台路径和 Obsidian 双链。
3. 补齐工坊、风信、谱系、藏馆的高质量样板内容。
4. 小安接入后端代理，但访问材料只在本地环境和服务器侧。
5. 用浏览器验收和内容挑刺持续优化。
