## 一、提示词工程操作模式

> 把提示词写成有角色、材料、边界、输出和验收的操作单。

提示词工程不应该被讲成神秘技巧。它就是一张工作单：你要做什么，读什么材料，不能碰什么边界，最后交付什么结果。

### 最小六块结构

1. **角色**：你现在扮演什么角色
2. **背景**：这件事为什么发生
3. **输入**：你可以使用哪些材料
4. **边界**：哪些内容不能使用、不能公开、不能猜
5. **输出**：结果要写成什么结构
6. **验收**：怎样判断这次输出合格

### 三个可复用模板

**模板一：资料改写**

```
你是一个公开资料库编辑。

背景：我有一段本地材料，需要改写成公开安全、中文小白能读懂的页面。

输入：只使用我提供的摘要，不要补充私人细节。

边界：不要写访问材料、账号、真实本机路径、私人聊天原文、未经证实的人身信息。
不确定的地方标记为"待复核"，不要当成事实。

输出结构：
1. 这是什么  2. 为什么重要  3. 真实操作历程
4. 小白复刻路径  5. 容易踩的坑  6. 下一步  7. 公开边界

验收：读者看完后应能说清它是什么、怎么开始、哪里不能乱做。
```

**模板二：项目复盘**

```
你是一个项目复盘教练。

请不要只写项目介绍。请说明：
- 最初想解决什么问题
- 做了哪些具体动作
- 用了哪些工具
- 哪一步最容易失败
- 小白怎样复刻一个最小版本
- 这个项目体现了什么学习方法
- 当前状态是否真实完成

输出要使用中文，先解释人话，再出现专业词。
```

**模板三：前端内容验收**

```
你是一个中文读者体验审稿人。

请检查这个页面是否适合小白：
1. 第一屏是否能看懂这里是谁的书房
2. 是否出现过多生硬术语
3. 项目卡是否说明了为什么值得看
4. 是否给出复刻路径
5. 是否有文学感、心理层、社会层和哲学层，但没有空话
6. 是否避免了访问材料、账号、本机路径和私密材料

请输出：必须修改的问题、可以增强的地方、保持不动的优点、最小修改建议。
```

### 文学、心理、社会、哲学怎样结合

- **文学感**：不是写得虚，而是让句子有画面、有节奏、有准确的对象
- **心理层**：解释学习中的压力、犹豫、焦虑和修正，不是贴标签
- **社会层**：说明工具、平台、读者、协作者和发布边界怎样影响选择
- **哲学层**：克制。只说清一个小观点——经验只有能被复刻、能被修正，才会从私人记忆变成公共知识

### 发布前检查

- 是否有具体项目或材料支撑
- 是否能被小白读懂
- 是否有操作步骤
- 是否写明风险和边界
- 是否遗漏来源
- 是否包含不该公开的信息

> 提示词不是终点。它只是把混乱材料带到第一版草稿的桥。

## 二、Agent 工作流入门

> 把智能体拆成输入、检索、模型、确认和输出五个节点。

Agent 是会按步骤工作的助手：先听问题，再决定要不要查资料，必要时调用工具，遇到危险动作先问人，最后给出结果。

### 最小五节点

1. **输入节点**：读者提出问题
2. **检索节点**：只查公开资料
3. **模型节点**：根据问题和资料组织答案
4. **确认节点**：凡是发布、写文件、调接口、高成本请求，都先停下来确认
5. **输出节点**：给答案，同时说明下一步

### 实际示例

小白问："我想做一个个人资料库，第一步是什么？"

流程：
1. 输入节点收到问题
2. 检索节点找"个人书房后端入门""AI 前端搭建入门""个人资料库平台"
3. 模型节点总结最小路线：先写一页公开内容，再生成数据，再做列表和详情页
4. 确认节点检查是否提到访问材料或本地细节
5. 输出节点给出三步行动

### 容易踩的坑

- 只做拖拽界面，不写节点规则
- 让 Agent 直接读未整理材料
- 没有人工确认节点
- 没有运行记录，出错后无法追踪
- 一开始设计太多角色，每个都很浅

### 三层理解

- **心理层**：人喜欢把复杂问题交给"万能助手"，但真正减少焦虑的，是把问题拆成可检查的步骤
- **社会层**：Agent 连接了模型平台、工具权限、知识来源和用户信任
- **哲学层**：可靠的智能不在于不停行动，而在于知道什么时候停下来

## 三、AI 小白复刻工作流

> 从看懂目标、拆解步骤、做最小版、验收结果到记录失败。

复刻不是照抄成品。复刻是理解一个成果为什么这样做，然后做出一个更小、更稳、能验收的版本。

### 五步复刻法

1. **看懂目标**：这件作品解决什么问题
2. **拆成步骤**：输入、动作、输出分别是什么
3. **做最小版**：只保留最关键的一条链路
4. **验收结果**：能打开、能搜索、能返回、能解释
5. **写下失败**：哪里卡住，为什么，下次怎么避开

### 例子：复刻安的书房前端

不要一开始做完整网站。先做：
1. 一个首页
2. 三张项目卡
3. 一个详情页
4. 一个搜索框
5. 一条检查清单

这就是最小版本——它小，但能证明你已经理解了"内容进入前端"的闭环。

### 复刻时的七个问题

1. 我在复刻什么，不复刻什么？
2. 我现在能拿到哪些公开材料？
3. 最小成果是什么？
4. 哪一步最容易失败？
5. 怎样知道我做对了？
6. 如果失败，先查数据、页面、接口还是提示词？
7. 这次留下了什么经验？

## 四、Codex 验证工作流

> 把结构、隐私、测试、站点数据和 Git 状态串成一条可复跑的公开维护验证链。

验证工作流不是单独一个工具，而是一种习惯——让每个"完成了"的声明都有检查可追溯。

### 标准检查项

- 结构和链接检查
- 隐私扫描
- 私有层隔离检查
- 公开清单生成
- 单元测试
- 前端构建和检查

### 复刻路径

1. 做一个小改动
2. 重新生成依赖的数据
3. 运行相关检查
4. 修问题，但不削弱规则
5. 记录到日志

### 公开边界

验证报告可以引用状态，但原始私有发现和敏感审查细节不能进入公开页面。

## 五、Agent 技能治理

> 把可复用流程写成技能，把私有提示词、本地状态和执行细节留在非公开层。

技能治理是把反复出现的 Agent 工作方式写成稳定的流程资产。解决的不是"怎样让 Agent 看起来更强"，而是"怎样让 Agent 的行为可复用、可检查、可交接"。

### 公开页面可以写

- 这个技能解决什么问题
- 什么时候使用，什么时候不要使用
- 它保护哪些边界
- 它会产出什么
- 怎样判断这次运行是合格的
- 它和资料库、前端、后端的关系

### 不能进公开页面的

- 原始提示词
- 本地记忆文件
- 私有会话记录
- 没有复核的旧规则
- 账号绑定和运行状态
- 能还原本机环境的路径、日志和配置

### 小白复刻六步

1. 选一个经常重复的任务
2. 写触发条件：什么时候应该用
3. 写输入材料：允许读什么
4. 写禁止事项：不能读什么、不能公开什么、不能猜什么
5. 写输出结构：结果要包含什么
6. 写验收方式：怎么证明它完成了

> 如果一个流程不能写出这六项，就先不要叫它技能。

技能不是给 Agent 戴头衔。技能是把混乱经验压缩成下一次能重复使用的秩序。

---

## 把它们串起来：一条完整的工作流

如果你是一个用 AI 做项目的人，完整流程是这样：

1. **写提示词** → 用六块结构写清楚要什么
2. **搭 Agent** → 五个节点串成可检查的流程
3. **做最小版** → 先做一个能验收的小成果
4. **验证结果** → 跑检查项，确认合格
5. **沉淀规则** → 把反复用的写成技能，下次直接触发

这五步循环起来，就是一条从"跟 AI 瞎聊"到"系统化产出"的路径。
